Теория к лабораторной работе:

XSD схемы

Вниз

 

В прошлой лабораторной работе мы разбирали DTD схемы. Следует отметить, что в настоящее время идет отказ от использования схем DTD по ряду причин, одной из которых является то, что схемы DTD нельзя выразить с помощью синтаксиса XML, что требует от разработчиков изучения нового языка программирования, использования специальных инструментов и т.д.

На смену DTD пришел стандарт консорциума W3C — XML Schema, называемый также XSD (XML Schema Definition).

 

Корневым элементом в схеме XML является элемент Schema, который содержит все остальные элементы в документе схемы. В рамках корневого элемента схемы XSD атрибутом xmlns определяется пространство имен XML Schema, которое содержит элементы и атрибуты XSD схемы.

<xsd:schema xmlns:xsd="http://www.w3.org/2001/XMLSchema">

Обратите внимание на то, что все элементы XSD начинаются с префикса xsd:, который указывается для пространства имен XSD, объявленного в корневом элементе экземпляра схемы.

 

В самом XML-документе, который проверяется с помощью схемы, должно содержаться  объявление пространства имен. Пространство имен — это именованная совокупность имён элементов и атрибутов, служащая для обеспечения их уникальности в XML-документе (подробнее см. здесь).

 Пространство имен всегда указывается в корневом элементе экземпляра документа с помощью атрибута xmlns:

xmlns:xsi=«http://www.w3.org/2001/XMLSchema-instance»

Это пространство имен содержит элементы и атрибуты XMLSchema, которые можно включать в документ XML. По общему соглашению префикс xsi используется для этого пространства имен и добавляется в начале имен всех элементов и атрибутов, принадлежащих пространству имен, отделяясь от них двоеточием.

 

Ссылка на конкретную схему приводится в атрибуте (в данном случае схема является локальным файлом)

xsi:noNamespaceSchemaLocation="имя_схемы.xsd"

 

В схемах XSD элементы, используемые в документах XML, бывают двух типов: простые (simpleType) и сложные(complexType). Элементы сложных типов могут содержать атрибуты, а также дочерние элементы. Элементы простых типов такими возможностями не обладают.

 

Язык XSD содержит большое количество встроенных простых типов данных. В дополнение встроенным типам данных XSD предоставляет возможность создания новых типов данных, основанных на встроенных. Автор схемы может создать подобные производные типы данных и использовать их в документе XSD. Полный список простых типов данных XSD вы можете найти по адресу http://www.w3.org/TR/xmlschema-0/.

Пример 1

Документ XML, содержащий элемент простого типа:

<?xml version="1.0" encoding="UTF-8"?>

<!--объявление пространства имен-->

<!--ссылка на схему XSD-->

<note xmlns:xsi="http://www.w3.org/2001/XMLSchema" xsi:noNamespaceSchemaLocation="simple.xsd">

Не забудь купить молоко по дороге с работы домой

</note>

 

Схема XSD, которая проверяет данный документ (simple.xsd):

<?xmlversion="1.0"?>

<!--объявление пространства имен-->

<!—объявляется и определяется элемент note с типом данных string с пространства имен XSD-->

<xsd:schema xmlns:xsd=”http://www.w3.org/2001/XMLSchema”>

<xsd:element name="note" type="xsd:string"/>

</xsd:schema>

 

Пример 2

Документ XML, содержащий элемент сложного типа (содержащий дочерний элемент):

<?xml version="1.0" encoding="UTF-8"?>

<!--элемент note содержит дочерний элемент message-->

<note

xmlns:xsi="http://www.w3.org/2000/10/XMLSchema"

xsi:noNamespaceSchemaLocation="complex.xsd">

<message>Не забудь купить молоко по дороге с работы домой</message>

</note>

 

Схема XSD, которая проверяет данный документ (complex.xsd):

<?xml version="1.0"?>

<xsd:schema xmlns:xsd=”http://www.w3.org/2001/XMLSchema”>

<xsd:element name="note">

<xsd:complexType>

<xsd:sequence>

<xsd:element name="message" type="xsd:string"/>

</xsd:sequence>

</xsd:complexType>

</xsd:element>

</xsd:schema>

 

Элемент sequence

Позволяет указать последовательность дочерних элементов.

<xsd:element name=”message” type=”xsd:string”/>

<xsd:element name=”note”>

<xsd:complexType>

<xsd:sequence>

<xsd:element ref=”message”/>

</xsd:sequence>

</xsd:complexType>

</xsd:element>

Ограничения вхождений в схемах XSD

В отличие от других языков описания схем, XSD позволяет вам определить количество вхождений элемента с определенной точностью. Вы можете задать минимальное и максимальное количество вхождений элемента с помощью атрибутов minOccurs и maxOccurs элемента xsd:element соответственно. На возможные значения этих атрибутов налагаются определенные ограничения:

minOccurs=”0”

вхождение элемента необязательно

minOccurs=”1”

предусмотрено одно вхождение элемента

maxOccurs=”1”

minOccurs и maxOccurs не указаны

minOccurs=”2 (3,4,…)”

минимальное число вхождений равно 2 (3,4,..)

maxOccurs=”2 (3,4,…)”

максимальное число вхождений равно 2 (3,4,..)

maxOccurs=”unbounded”

число вхождений не ограничено

 

Атрибуты в схемах XSD

Объявление атрибутов в схемах XSD очень похоже на объявление элементов, за исключением того, что атрибуты объявляются с помощью объявлений attributes, a не element.

Пример 3.

Документ:

<?xml version="1.0" encoding="UTF-8"?>

<note

xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"

xsi:noNamespaceSchemaLocation="3.xsd">

<message number="10" date="2009-10-29" from="Ольга Петрова">

Не забудь купить молоко по дороге с работы домой

</message>

</note>

 

Схема XSD (3.xsd):

<?xml version="1.0" encoding="Windows-1251"?>

<xsd:schema xmlns:xsd="http://www.w3.org/2001/XMLSchema">

<xsd:complexType name="messageType">

<xsd:attribute name="number" type="xsd:integer" use="required"/>

<xsd:attribute name="date" type="xsd:date" use="required"/>

<xsd:attribute name="from" type="xsd:string" use="required"/>

</xsd:complexType>

<xsd:element name="note">

<xsd:complexType>

<xsd:sequence>

<xsd:element name="message" type="messageType"/>

</xsd:sequence>

</xsd:complexType>

</xsd:element>

</xsd:schema>

Анализ

Обратите внимание, что элементу complexType присвоено имя “messageType”. Ниже объявление элемента message содержит ссылку (type=”messageType”). Для чего это нужно? Дело в том, что элемент complexType является элементом только для XSD схемы, но не для XML документа, созданного на основании этой схемы (заметьте, в документе элемента complexType нет). Задавая элемент complexType, мы создам сложный тип под названием “messageType”, и этот тип содержит 3 атрибута со своими характеристиками. Но complexType не является типом какого-либо определенного элемента, он сам по себе. А далее, когда мы характеризуем  уже непосредственно элемент XML документа (в данном случае это message), то ссылаемся на complexType, тем самым объявляя, что элемент message будет содержать 3 атрибута с характеристиками, заданными в complexType. Если впоследствии в данном XML документе мы захотим добавить еще какой-либо элемент, содержащий такие же 3 атрибута с такими же характеристиками, в XSD схеме нам достаточно будет при объявлении элемента сделать ссылку type=”messageType”.

 

И напоследок рассмотрим еще один пример:

 

Пример 4.

Документ:

<?xml version="1.0" encoding="UTF-8"?>

<book

xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"

xsi:noNamespaceSchemaLocation="4.xsd" id="1">

<author>Толстой Лев Николаевич</author>

<title>Война и мир</title>

<price>80 тыс. бел. руб.</price>

</book>

 

Схема XSD (4.xsd):

<?xml version="1.0" encoding="Windows-1251"?>

<xsd:schema xmlns:xsd="http://www.w3.org/2001/XMLSchema">

<xsd:element name="book">

<xsd:complexType mixed="true">

<xsd:sequence>

<xsd:element name="author" type="xsd:string"/>

<xsd:element name="title" type="xsd:string"/>

<xsd:element name="price" type="xsd:string"/>

</xsd:sequence>

<xsd:attribute name="id" type="xsd:integer" use="required"/>

</xsd:complexType>

</xsd:element>

</xsd:schema>

 

Анализ

В данном пример элемент сложного типа note содержит 3 простых элемента (author, title, price) и 1 атрибут (id). Обратите внимание на выражение в XSD схеме: mixed=”true”. Такой атрибут используется в том случае, если элемент сложного типа имеет смешенное содержимое (в данном случае это смешенное содержимое представлено тремя элементами и атрибутом. Могут быть случаи, когда элемент, например, содержит непосредственно свое значение и дочерние элементы, или другие случаи).
